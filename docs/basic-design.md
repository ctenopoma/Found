# 基本設計書 — フォルダ管理・全文検索システム

## 1. 目的と背景

社内の共有フォルダは案件ごとにフォルダを作成しているが、プロジェクトごとに
フォルダの作り方がまちまちで、管理・検索アプリケーションを開発しづらい。

本システムは以下を提供する。

1. **フォルダテンプレート**を定義し、テンプレートに従ってフォルダ構造を標準化する
2. テンプレート準拠フォルダを **Fess** でクロールし、全文検索を可能にする
3. ファイルサーバの **Windows AD / ACL 権限**に従って、検索結果を閲覧可能なユーザーにのみ表示する

本システムの本質的な課題は「フォルダ管理」よりも、
**AD 認証と権限の境界を Linux サーバ上で成立させられるか**にある。

## 2. 全体構成

```
[Windowsクライアント (ADユーザー)]
        │ ① ブラウザ / Tauri アプリ → アプリへのログイン認証 (SSO)
        ▼
[アプリサーバ (Linux)]  REST API + 管理画面
        │ ② サービスアカウントで SMB アクセス (フォルダ生成)
        │ ⑤ Fess REST API でクロール制御・検索
        ▼
[Fess (全文検索 / Linux + OpenSearch)]
        │ ③ SMB/CIFS クロール + ACL 取得 + 権限フィルタ
        ▼
[ファイルサーバ (Windows SMB / AD認証・ACL)]
```

### 認証の3つの境界（最重要）

| 番号 | 境界 | 内容 | 検証 |
|---|---|---|---|
| ① | クライアント → アプリ | LDAP 認証 + memberOf でグループ/SID 取得 | 検証C |
| ② | Linux → SMB | サービスアカウントでファイル読み書き | 検証B / D |
| ③ | 検索結果の権限制御 | ファイル ACL とユーザーグループの突合 | **検証A（最重要）** |

「Linux から AD で認証できるか」は一見1つの問いだが、実際は①②③の別問題である。

## 3. コンポーネント

| コンポーネント | 役割 | 技術 |
|---|---|---|
| アプリサーバ | テンプレート管理、フォルダ生成、Fess 連携、管理画面/API | Linux 上の Web アプリ |
| Fess | クロール、全文検索、権限フィルタ | Fess + OpenSearch |
| DB | テンプレート定義、案件メタデータ、監査ログ | RDB |
| ファイルサーバ | フォルダ/ファイル実体 | Windows SMB 共有 (AD認証, NTFS ACL) |
| AD | ユーザー/グループ認証基盤 | **オンプレAD（LDAP サーバあり）** |

## 4. 機能設計

| 機能 | 概要 |
|---|---|
| テンプレート定義 | 案件種別ごとのフォルダ構造雛形を登録（例：`01_見積/02_契約/03_納品`） |
| フォルダ生成 | テンプレートに従い共有フォルダ上に実フォルダを作成。命名規則を強制 |
| クロール連携 | テンプレート準拠フォルダを Fess のクロール対象に登録、スケジュール実行 |
| 検索 | 全文検索 + メタデータ（案件名・種別・作成日）ファセット検索 |
| 権限制御 | ファイル ACL に従い、閲覧可能ユーザーにのみ検索結果を表示 |
| 監査 | フォルダ生成・テンプレート変更の操作ログ |

現状課題「フォルダの作り方がまちまち」への解は **テンプレート強制による構造標準化**であり、
検索はその上に乗る付加価値という位置づけ。

## 5. 認証設計

### ① クライアント → アプリ（ユーザー認証）
**前提：オンプレ AD（LDAP サーバあり）。Azure AD / AD FS は無し。**
- 主軸：**LDAP（LDAPS）認証**。アプリ・Fess とも AD の LDAP に bind してユーザー認証し、
  `memberOf` から所属グループ（および SID）を取得する
- Fess は標準で LDAP 認証に対応（管理画面で LDAP プロバイダを設定）
- SSO を求める場合：**Kerberos / SPNEGO**（ドメイン参加 PC + アプリサーバの SPN 登録）。
  まずは LDAP の ID/パスワード認証で成立を確認し、SSO は後追いで可
- Tauri も内部の認可フローで同じ LDAP/Kerberos 認証を利用

### ② Fess / アプリ → ファイルサーバ
- **AD サービスアカウント**で SMB/CIFS 共有にアクセス
- Kerberos 認証（時刻同期 = NTP 必須）。NTLM はフォールバック

### ③ 検索結果の権限制御（最難所）
- Fess が SMB クロール時にファイルの **許可 SID（ACL）**をインデックスに格納
- 検索時、ログインユーザーの **AD グループ SID** と突合し、閲覧可能ファイルのみ返却
- ②（ファイルを読めるか）と③（ACL を取得し突合できるか）は**別問題**

## 6. 配信方式（Web / Tauri）

ビジネスロジックは**アプリサーバ API に集約**し、フロントは薄くする。

| 方式 | 位置づけ |
|---|---|
| Web 版 | ブラウザ。SSO と相性良、導入障壁最小。**PoC・初期展開はこちら** |
| Tauri 版 | Rust + WebView。ローカルファイル操作・OS 統合が必要な場合に価値。認証は OIDC を内蔵ブラウザで処理。後追いで可 |

指針：まず Web API を固め UI を共有 → Tauri は後追い。

## 7. リスクと留意点

- **権限フィルタ（③ / 検証A）が本命リスク**。SMB ACL → AD グループの自動マッピングは環境依存
- **Kerberos / 時刻同期**：Linux⇔AD は時刻ズレに敏感。NTP 必須
- **テンプレート移行方針**：既存フォルダを強制移行するか、新規案件のみ適用するか業務側と合意
- **OSS 保守**：Fess のバージョン追従、SSO 周りの運用ナレッジを社内に残す

## 8. 確定事項 / 未確定事項

確定：
- AD の形態 = **オンプレ AD（LDAP サーバあり）**。認証は LDAP(S) 主軸、SSO は Kerberos/SPNEGO で後追い

未確定（要確認）：
- LDAPS（636/TLS）の可否と CA 証明書の入手経路（社内 CA か）
- ベース DN / バインド用サービスアカウント、グループの `memberOf` 運用（ネストグループの有無）
- ファイルサーバの SMB バージョン（SMB2/3）
- 既存フォルダの移行範囲
